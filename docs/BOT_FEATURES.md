# NutriSnap — фичи бота и команды

Что умеет NutriSnap-бот, какие команды доступны и какие способы ввода еды поддерживаются. Документ — спека для разработки и источник для help-текста / описания бота в BotFather.

> Статус каждой фичи: **✅ готово** — работает в проде; **🚧 в работе** — частично; **📋 план** — описано, ещё не реализовано.

---

## Что бот умеет (one-liner)

> Дневник питания с AI-вводом: сфоткал тарелку или ингредиенты на весах — получил КБЖУ и записал приём пищи. Без ручного поиска продуктов и без копипасты в FatSecret.

Главные сценарии:

1. **Быстрый лог приёма пищи** — фото / голос / текст → один тап «обед / ужин / перекус».
2. **Рассчитать свою норму** — онбординг по формуле Mifflin-St Jeor (пол, вес, рост, возраст, активность, цель).
3. **Сохранить домашний рецепт** — раз посчитал КБЖУ блюда «курица с морковью» → дальше тапаешь «150 г» и готово.
4. **Дневник в Mini App** — графики, история, изменение профиля и норм.

## Команды бота

| Команда | Что делает | Статус |
|---|---|---|
| `/start` | Регистрирует юзера. Для новых — **обязательно** запускает онбординг (без него фото/голос/текст не обрабатываются). Для вернувшихся — короткое приветствие + кнопки «Открыть дневник» и «Пересчитать норму». | ✅ |
| `/onboard` | Пройти онбординг заново (пересчитать суточную норму КБЖУ). | ✅ |
| `/calculate` | Калькулятор КБЖУ блюда по ингредиентам. ConversationHandler: COLLECTING → NAMING → сохранить в базу или только посчитать. Перехватывает пересланные списки до общего текст-хендлера. | ✅ |
| `/recommend [запрос]` | AI-нутрициолог: LangGraph-subgraph анализирует дефицит БЖУ за день, делает Qdrant top-20 по каталогу, LLM выбирает 3 с обоснованием. Free-form запрос опционален. | ✅ |
| `/cancel` | Прервать любую активную конверсию (онбординг, recipe builder, калькулятор). | ✅ |
| `/help` | Сводка команд и способов ввода. | ✅ |
| `/open` | Открыть Mini App кнопкой `WebApp`. | ✅ |
| `/today` | Сводка КБЖУ за сегодня прямо в чате. | 🚧 заглушка |
| `/week` | Сводка КБЖУ за неделю. | 🚧 заглушка |
| `/remind` | Настройка умных напоминаний (по времени или по контексту: «не ел 5 ч»). | 📋 план |

> «Заглушка» — команда отвечает «скоро» и не падает, но реальной агрегации пока нет.

## Способы ввода еды

Всё что не команда и не часть онбординга — обрабатывается LangGraph'ом. Юзер выбирает приём пищи (завтрак / обед / ужин / перекус) inline-кнопкой после результата.

### 1. Текст ✅
> «съел 200 г куриной грудки и салат с помидорами»

Парсер → lookup КБЖУ → подтверждение.

### 2. Голосовое сообщение ✅
Whisper транскрибирует → тот же текстовый пайплайн.

### 3. Фото блюда ✅
GPT-4o-vision определяет состав и порции → lookup → подтверждение.

### 4. Пересланное сообщение ✅
Текст из чужого сообщения (например, рецепт от друга, состав с упаковки) → парсер.

> **Важно:** пересланные сообщения вне активного `/calculate` логируются как обычный приём пищи. Если нужно накопить список ингредиентов — используй `/calculate`; ConversationHandler перехватит пересланные строчки и НЕ создаст отдельные приёмы.

### 5. Disambiguation top-3 (уточнение продукта) ✅

При однозначном текстовом/голосовом вводе единственного продукта:

1. Граф резолвит КБЖУ через MCP как обычно.
2. Бот делает Qdrant top-3 поиск по имени продукта.
3. Если оценки близки (spread ≤ 0.20, min score ≥ 0.50) — вместо meal-type кнопок показываются 3 варианта продукта.
4. Юзер выбирает нужный → пересчёт порции → meal-type keyboard.

Не показывается: для фото, для приёмов с несколькими продуктами, когда одна позиция явно лидирует.

### 6. Альбом / медиагруппа из фото 📋
Telegram присылает альбом как N разных update'ов с одним `media_group_id`. Сейчас бот отвечает N раз — баг.

**План:** дебаунс ~1 с через `Application.job_queue.run_once`, копим фото в `context.bot_data[f"album:{mgid}"]`, обрабатываем пачкой как одну сцену с несколькими ракурсами / несколькими блюдами.

Также применить тот же паттерн к multi-forward (тишина 1.5 с → flush).

### 7. Recipe builder — фото ингредиентов на весах + название блюда ✅

**Самый частый и самый бесящий сценарий**: юзер готовит дома, фоткает каждый ингредиент на кухонных весах, потом вручную перегоняет ChatGPT'ом в КБЖУ и копипастит в FatSecret.

**Флоу:**

1. Юзер шлёт фото (одно или альбом) ингредиентов на весах + подпись «курица с морковью».
2. Бот распознаёт каждый ингредиент с граммовкой со шкалы весов.
3. Под результатом появляется кнопка **🍳 Это рецепт — сохранить**.
4. Если юзер тапнул эту кнопку — запускается `RecipeBuilderConversation`:
   - Состояние **COLLECTING** — бот принимает ещё фото ингредиентов (если юзер забыл что-то взвесить), накапливает.
   - Юзер тапает кнопку **✅ Готово** когда всё перечислил.
   - Состояние **AWAIT_TOTAL_WEIGHT** — бот спрашивает: *«Сколько весит готовое блюдо в граммах? (взвесь сковороду/кастрюлю минус её вес)»*
   - Юзер пишет «320» → пересчёт КБЖУ на 100 г готового блюда.
   - Бот сохраняет в `foods` как `source=USER_RECIPE`, `created_by_user_id=user.id`, `aliases=["курица с морковью"]`, метрика `g`, КБЖУ на 100 г.
   - Состояние **AWAIT_PORTION** — бот спрашивает: *«Какую порцию съел сейчас?»*
   - Юзер пишет «150» → лог приёма пищи как обычно (с выбором meal_type).
5. **В следующий раз** «курица с морковью» уже найдётся локально (PostgreSQL → Qdrant RAG). Юзер шлёт фото блюда или просто пишет «150 г курицы с морковью» → один тап.

**Триггер:** кнопка под результатом обычной обработки фото (явный, предсказуемый). Автодетект на основе содержимого («вижу весы, добавлю кнопку») — возможен, но НЕ полагаемся только на него.

**Финализация:** юзер сам нажимает «✅ Готово», бот не угадывает.

**Что нужно построить:**
- `ConversationHandler` с тремя состояниями (COLLECTING / AWAIT_TOTAL_WEIGHT / AWAIT_PORTION)
- Кнопка `🍳 Это рецепт — сохранить` под обычным результатом фото (callback `recipe:start:{draft_id}`)
- Vision-промпт под весы: structured output `[{name, weight_g, source: "scale"|"estimate"}]`
- Расширить `FoodSource` enum значением `USER_RECIPE` (`models.py`)
- Repository метод `save_user_recipe(user, name, total_kcal, total_protein, total_fat, total_carbs, cooked_weight_g)` — пересчёт на 100 г и upsert в `foods`
- Этот кейс **зависит от (5) album batching** — рецепт почти всегда альбом

## Запланированные фичи (USP — «AI nutritionist, not bookkeeper»)

См. `docs/VALUE_PROPOSITION.md`. Три killer-фичи к защите:

### Smart reminders 📋
- *«Утром в 9:00, если не ел — напомни про завтрак»*
- *«Прошло 5 ч без приёма пищи — спроси что ел»*
- Конфигурится через `/remind` или в Mini App. Хранится в `users.reminders_config` (JSONB).

### AI nutritionist — `/recommend` 📋
- LangGraph агент берёт сегодняшний дневник + цель юзера → рекомендует «не добираешь 30 г белка — съешь греческий йогурт или 100 г тунца».
- Использует MCP-tool `get_daily_summary` + RAG по продуктам.

### Variety nudges 📋
- Анализирует историю последних 7–14 дней. Если юзер ест одно и то же — предлагает альтернативу с похожим КБЖУ-профилем.
- Пример: *«Куриная грудка пятый день подряд. Хочешь, посчитаю КБЖУ телятины / рыбы / индейки?»*

## Зависимости и порядок реализации

```
album batching (case 5)
   ↓
recipe builder (case 6)         ←  главный USP
   ↓
/today, /week (real summaries)
   ↓
/recommend (AI nutritionist)    ←  killer feature 2
   ↓
smart reminders + variety nudges  ←  killer features 1 и 3
```

Recipe builder делается **до** killer features 1/3, потому что он закрывает самую конкретную и понятную пользовательскую боль и сильнее всего отличает NutriSnap от FatSecret.
